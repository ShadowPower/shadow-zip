# 产品需求文档

文档日期：2026-05-07

## 1. 产品定位

本产品是一款面向 Windows、Linux 与 macOS 的跨平台图形归档工具。它应当提供压缩、解压、归档浏览、图片预览、任务管理、密码处理、冲突处理和平台集成能力，并在体验层面向 Bandizip 这类成熟桌面归档软件靠近。

产品的核心价值不在于简单调用压缩库完成文件转换，而在于使用户能够快速理解归档内容、可靠地处理大文件和大量小文件、在任务执行过程中保持界面可响应，并在面对复杂格式时获得清晰、可解释的反馈。对于普通用户，产品应表现为一个稳定、快速、少打扰的桌面工具；对于开发者和高级用户，产品还应可靠处理 tar.gz、tar.xz、tar.zst、7z 扩展 codec、加密包、分卷包、Unix 权限和符号链接等复杂场景。

由于目标平台覆盖三大桌面系统，产品必须区分“核心能力”和“平台集成能力”。打开、浏览、解压、压缩、预览、任务队列和缓存管理属于核心能力，必须在三平台上统一提供。右键菜单、文件关联、拖出到系统文件管理器、通知和自动更新属于平台集成能力，应在不影响核心功能稳定性的前提下分阶段实现。

## 2. 用户与场景

普通桌面用户的典型需求是快速打开归档、拖拽压缩、解压到指定目录、处理密码包、处理文件覆盖冲突，以及在图片归档中直接查看内容。此类用户不会关心底层 codec，但会非常敏感于界面是否卡顿、提示是否清楚、操作是否符合桌面软件习惯。

开发者用户通常会处理 tar.gz、tar.xz、tar.zst、zip、7z 等格式，并关心 Unix 权限、符号链接、时间戳、长路径和大量小文件的处理。对于这类用户，tar.xx 是否单遍处理非常关键。如果软件先生成中间 tar 再二次解包，即使功能正确，也会在大包场景下显著浪费磁盘和时间。

高级归档用户会关注压缩方法、压缩等级、solid 选项、加密、分卷、ZIP AES、7z header encryption，以及 7z + Zstandard、7z + LZ4、7z + Brotli 等现代或扩展组合。这类用户需要明确的能力呈现，也需要产品在创建归档时说明兼容性代价。

## 3. 功能范围

第一阶段产品应覆盖主线归档工作流。用户必须能够打开、浏览、解压和创建 ZIP、7z、tar、tar.gz、tar.xz、tar.zst。ZIP 应支持 ZIP64、AES、分卷和常见压缩方法；7z 应支持 LZMA/LZMA2，并正式纳入 ZSTD、LZ4、Brotli 等扩展 codec 的读取能力；tar.xx 必须采用单遍流式压缩和解压；图片预览应覆盖 JPEG、PNG、WebP、GIF、BMP、TIFF、ICO、AVIF 等常见格式的基础预览体验。

第二阶段应补齐复杂格式和复杂交互。RAR/RAR5 的列表、测试和解压应作为内建能力提供；tar.xx 应具备索引缓存，以改善二次打开和浏览体验；solid 7z 的单文件预览应通过状态提示、邻近预取和任务取消机制降低等待感；文件关联和基础系统通知应进入可用状态。

第三阶段才适合处理深度平台集成和授权复杂能力。Windows Explorer 右键菜单、macOS Finder 集成、Linux 文件管理器扩展、从归档拖出文件、自动更新、崩溃恢复和 RAR 创建都属于这一阶段。尤其是 RAR 创建，它不是单纯技术实现问题，还涉及 RARLAB 授权、分发和商业条款，应以外部组件或授权能力的方式设计。

## 4. 性能与响应目标

本产品必须将响应速度作为需求，而不是后期优化事项。主窗口冷启动到可交互的目标应控制在 500 ms 内，热启动目标应控制在 250 ms 内。普通 ZIP 打开后应在约 200 ms 内显示目录骨架；普通 7z 应在约 300 ms 内显示基本 listing，但 solid、加密 header 或复杂分卷归档可以例外。图片预览应先显示 metadata，再生成缩略图，普通 ZIP 内图片的首张缩略图目标应控制在 150 ms 左右。

这些指标不是对所有硬件和所有归档的绝对承诺，而是设计约束。实现上必须保证 UI 线程不执行归档扫描、压缩、解压、图片解码、图片缩放或大量文件系统遍历。所有长任务应进入任务系统，并通过节流后的进度事件更新界面。进度刷新应控制在约 30 至 60 ms 聚合一次，避免大量小文件场景下因 UI 消息过密导致界面卡顿。

## 5. 资源占用要求

内存使用应保持与当前操作相关，而不应与归档总大小线性相关。大型归档的 listing 应采用紧凑 metadata，文件列表必须虚拟化渲染，路径字符串应尽量共享或压缩存储。图片预览必须设置输入大小、输出像素、帧数和缓存总量限制。建议小文件内存预览阈值设为 8 MB 至 16 MB，全局缩略图缓存默认设为 64 MB 至 128 MB，并允许用户调整。

磁盘占用同样是产品体验的一部分。tar.gz、tar.xz、tar.zst 在解压和创建时不得默认生成中间 .tar。图片预览默认不应落盘，只有在系统外部查看器、拖出到文件管理器、大图高分辨率预览或解码器强依赖 seek 的情况下，才允许使用临时文件。所有临时文件和缓存必须位于专用目录，受容量上限管理，并提供可见的清理入口。

## 6. 交互要求

打开归档时，产品应尽快呈现目录骨架，并异步补充详细信息。ZIP 和非 solid 7z 应提供近似随机访问体验；solid 7z、solid RAR 和 tar.xx 则应明确显示其顺序读取或压缩块定位状态。产品不能把格式固有限制伪装成普通等待，否则用户会把合理延迟理解为程序卡死。

解压前必须进行目标目录、写权限、可用空间、路径冲突、危险 entry 和符号链接策略检查。冲突处理应采用批量面板，不应逐文件弹窗。密码输入应支持重试和会话级临时记忆，但默认不持久保存密码，也不得在日志中记录密码。

图片预览是本产品的重要差异化体验。用户选择图片时，右侧面板应先显示文件名、大小、尺寸、方向和色彩等 metadata，再加载缩略图和适应窗口预览。用户放大到 100% 或更高比例时，系统再按需生成更高分辨率结果。预览应支持空格打开、左右切换、滚轮缩放、双击 100% 与适应窗口切换，并在加载失败时显示结构化错误。

## 7. 安全要求

归档工具必须默认安全。所有解压路径都必须经过清洗，并验证最终写入路径仍位于目标目录内。包含 ../、绝对路径、Windows 盘符路径、UNC 路径、设备路径或混合分隔符绕过的 entry 必须被标记并阻止。符号链接默认应采用保守策略，除非用户明确选择跟随符号链接，并且系统仍需阻止其写出目标目录。

产品还需要防御归档炸弹和图片解码炸弹。实现应检测异常压缩比、异常解压体积、过多 entry、过深目录、超长路径、递归归档、超大图片、过多动图帧和异常内存分配。外部 helper 调用必须通过参数数组传递，不得拼接 shell 字符串，并应具备超时、取消、输出限制和版本可见性。

## 8. 验收标准

MVP 的验收不应只看功能是否“能跑”。更合理的验收标准是：三平台可以打开主窗口；ZIP、7z、tar.zst 的主线打开和解压流程稳定；7z + ZSTD 样本可被正确读取；tar.zst 解压过程中没有中间 tar 文件；图片预览不会突破内存上限；长任务执行期间窗口仍可交互；错误能够被结构化呈现；缓存可以清理；所有危险路径都被阻止。
