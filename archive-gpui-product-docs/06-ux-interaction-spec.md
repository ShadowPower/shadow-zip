# 交互规格说明书

文档日期：2026-05-07

## 1. 交互设计目标

本产品属于高频工具型桌面软件。它的界面应服务于重复操作、快速判断和稳定反馈，而不是展示性视觉表达。用户打开产品时，第一屏就应当是可操作的归档工作台；用户拖入文件、双击归档、选择图片或启动解压任务时，界面应立即回应，并在后台继续完成耗时工作。

交互设计必须尊重归档格式本身的差异。ZIP 和非 solid 7z 可以提供较强的即时浏览体验；tar.gz、tar.xz、tar.zst 需要顺序扫描；solid 7z 和 solid RAR 的单文件读取可能需要解码同一压缩块中的前序数据。产品应将这些差异转化为清楚的状态提示，而不是让用户面对无解释的等待。

## 2. 主窗口

主窗口建议采用工具型布局。顶部为稳定工具栏和搜索入口，左侧为目录树，中部为文件列表，右侧为预览与详情面板，底部为任务中心和状态栏。该布局能够同时满足浏览、选择、预览和任务观察，不需要用户频繁切换页面。

顶部工具栏应承载打开、解压、压缩、测试、搜索、视图切换和设置等高频命令。图标按钮应配备 tooltip，重要命令可以使用图标加文字。工具栏不应频繁变化，否则会降低工具型软件的可预测性。

文件列表是主操作区，必须具备虚拟化能力。它应支持多选、排序、过滤、右键菜单、复制路径和拖出。列信息应包含名称、大小、压缩后大小、类型、修改时间、压缩方法、加密状态和路径。对于大型归档，列表不应一次性创建所有行的完整 UI 对象，而应只渲染可见区域。

右侧预览面板应根据文件类型变化。图片文件显示 metadata、缩略图和预览图；文本文件显示只读文本预览；其他文件显示文件详情和外部打开入口；不支持预览时应说明原因。预览面板不应因为加载失败而空白，也不应因为大图解码阻塞主窗口。

## 3. 打开归档

用户可以通过双击归档、文件对话框、拖拽或最近文件打开归档。打开行为应分阶段呈现。对 ZIP，系统读取 central directory 后即可显示目录骨架，并在后台补充部分 metadata。对 7z，系统应在打开后识别 codec、filter、solid、加密和分卷信息。对 tar.xx，系统应进入流式扫描状态，并增量显示已扫描出的目录内容。

如果归档是 solid 7z，界面应在详情区或状态栏显示其访问特征，说明单文件提取和预览可能较慢。如果归档采用 header encryption，系统应在输入密码前说明目录信息可能不可见。如果归档使用不支持的 codec 或 filter，错误信息必须包含 codec 或 filter 名称，并说明是否正在尝试兼容后端。

## 4. 解压流程

解压交互应强调预检查和批处理。用户选择解压目标后，系统应检查目标目录、写权限、可用空间、路径冲突、危险路径和符号链接策略。若存在覆盖冲突，应打开批量冲突处理面板，而不是对每个文件弹窗。

任务执行期间，底部任务中心应显示当前阶段、当前文件、已处理大小、总大小、已处理文件数、总文件数、速度、ETA 和取消入口。ETA 在采样不足时不应给出不稳定数字，可以显示为正在估算。取消任务后，界面应进入取消中状态，直到后端到达安全取消点并完成清理。

完成后，任务中心应允许用户打开目标目录、查看错误详情、复制任务摘要或重试失败项。若部分 entry 因安全策略被阻止，完成状态应明确显示为“部分完成”，并列出被阻止原因。

## 5. 压缩流程

创建归档窗口应提供输出路径、格式、压缩方法、压缩等级、solid、加密、分卷、路径结构、符号链接策略和完成后操作。默认值应以兼容性为主。ZIP 默认 Deflate，7z 默认 LZMA2，tar.zst 默认 Zstandard 标准等级。高级 codec 如 7z + ZSTD、7z + LZ4、7z + Brotli 应放在高级选项中，并提示旧工具兼容性风险。

压缩任务应先进行输入扫描和体积估算，再进入实际写入阶段。大量小文件场景下，界面应显示文件数进度和字节进度两个维度，因为只看字节进度会让用户误判任务是否停滞。

## 6. 图片预览

图片预览应采用分级加载。用户选中图片后，系统先读取 metadata，包括文件名、大小、尺寸、方向、EXIF 和色彩信息；然后生成缩略图；再生成适应窗口预览；只有用户放大到 100% 或请求高质量查看时，才生成更高分辨率结果。

预览交互应支持空格打开或关闭预览，左右键切换上一张和下一张，滚轮缩放，双击切换 100% 和适应窗口。对于连续看图场景，系统可以对邻近图片预取 metadata 和低清图，但预取必须受优先级和缓存上限约束，不能挤占正在执行的解压任务。

对于 solid 7z、solid RAR 和 tar.xx 中的图片，预览可能需要顺序读取或定位压缩块。界面应显示正在定位、正在读取或正在解码的状态，并允许用户取消。这样可以将不可避免的格式成本转化为可理解的等待。

## 7. 密码与冲突

密码输入应简洁但安全。弹窗应显示归档名称和需要密码的范围，支持显示密码切换、重试和会话级临时记忆。默认不应持久保存密码，日志也不得记录密码。对于 header-encrypted 7z，密码弹窗应说明目录信息需要解密后才能显示。

冲突处理必须批量化。面板应展示冲突数量、当前冲突项、源文件信息和目标文件信息，并允许用户选择覆盖、跳过、自动重命名或仅保留较新文件。用户可以将策略应用于后续全部冲突。该设计能够避免大量文件解压时出现反复弹窗。

## 8. 错误呈现

错误提示应正式、明确且可操作。密码错误、不支持 codec、归档损坏、磁盘空间不足、权限不足、路径过长、路径穿越、符号链接策略阻止和外部 helper 不可用都应有不同文案。错误详情中可以包含后端名称和技术代码，但主提示应使用用户能够理解的语言。

对于 fallback 场景，界面不应给用户制造不确定感。例如，主后端无法处理某个 7z filter 时，可以显示正在使用兼容模式打开；若兼容模式仍失败，则显示不支持的 filter 名称和建议操作。

## 9. 设置

设置页面应覆盖默认解压目录、默认压缩格式、默认压缩等级、缓存大小、缓存清理、图片预览最大像素、密码会话记忆、高级 codec 显示、外部 helper 路径和日志级别。设置项应保持稳定，不应把调试选项暴露给普通用户。高级设置可以折叠，但不能隐藏影响兼容性和安全性的关键选项。
