# Bandizip 格式支持缺口清单

文档日期：2026-05-18

## 1. 文档说明

本文档用于把 Shadow Zip 与 Bandizip 的格式覆盖差距整理为可勾选的工程清单。清单排除自解压格式，例如 `EXE(sfx)`，因为自解压涉及可执行文件解析、执行风险提示和平台安全策略，应单独评估。

本文档只描述格式支持缺口和验收项，不替代 [03-format-support-matrix.md](./03-format-support-matrix.md) 中对 ZIP、7z、tar、RAR 等主线格式能力等级的定义。若某个格式可以通过现有后端顺带打开，也必须在产品能力模型、文件关联、测试样本和错误提示中形成明确承诺后，才视为正式支持。

参考基线为 Bandizip 官方格式列表：压缩支持 `ZIP`、`7Z(lzma2)`、`ZIPX(xz)`、`TAR`、`TGZ`、`LZH(lh7)`、`ISO(joliet)`、`GZ`、`XZ`；解压支持 40+ 格式。本文档同时列出 Shadow Zip 当前已支持的格式和仍缺失的格式，已支持项使用勾选框标记。

## 2. 通用验收项

每个新增格式至少应满足以下共同验收项。若某格式天然不支持某项，应在能力模型中显式标为不支持，而不是静默失败。

- [ ] 扩展名探测。
- [ ] 魔数或结构探测，避免只依赖扩展名。
- [ ] `ArchiveFormat` 或等价格式标识。
- [ ] `ArchiveCapabilities` 能准确表达列表、解压、创建、加密、分卷、随机访问和预览能力。
- [ ] 列目录包含路径、目录/文件类型、原始大小、压缩大小、修改时间、压缩方法、加密标记。
- [ ] 全部解压。
- [ ] 选中条目解压。
- [ ] 路径穿越、绝对路径、Windows drive path、UNC、device path 和符号链接策略检查。
- [ ] 密码需求、错误密码、损坏归档、不支持 codec、不支持 filter、截断数据的结构化错误。
- [ ] `test` 或等价完整性校验。
- [ ] CLI、app-core、后端单元测试和真实样本集成测试。
- [ ] 文件关联和用户可见格式名称。
- [ ] fallback 后端诊断日志，包含原后端、fallback 后端、失败原因和最终结果。

## 3. 流式处理强制要求

`tar.gz`、`tar.xz`、`tar.zst`、`tar.bz2`、`tar.lzma`、`tar.Z`、`tar.lz` 等组合格式必须实现全流式压缩和全流式解压。它们不能被实现为“两次操作”，也不能在默认路径中先生成中间 `.tar` 文件再继续处理。

解压的正确管线是：

```text
archive file -> compression decoder -> tar reader -> safe writer -> destination
```

创建的正确管线是：

```text
input scanner -> tar writer -> compression encoder -> archive file
```

强制验收项：

- [ ] 解压 `tar.gz` 时不得落盘生成中间 `.tar`。
- [ ] 解压 `tar.xz` 时不得落盘生成中间 `.tar`。
- [ ] 解压 `tar.zst` 时不得落盘生成中间 `.tar`。
- [ ] 创建 `tar.gz` 时不得先生成中间 `.tar`。
- [ ] 创建 `tar.xz` 时不得先生成中间 `.tar`。
- [ ] 创建 `tar.zst` 时不得先生成中间 `.tar`。
- [ ] 后续新增 `tar.bz2`、`tar.lzma`、`tar.Z`、`tar.lz`、`tar.br` 也必须遵守同一管线。
- [ ] 进度统计必须基于流式读写推进，不能依赖中间文件大小。
- [ ] 取消任务时必须能中断 decoder、tar reader/writer 和目标写入。
- [ ] 峰值内存必须与 buffer、单条目策略和预览限制相关，不能随归档总大小线性增长。
- [ ] 临时目录占用不得接近未压缩 tar 总大小。
- [ ] 性能测试必须验证大包、大量小文件、取消、磁盘空间不足和损坏压缩流。

## 13. 优先级建议

建议按用户价值和实现风险分批推进。

- [ ] 第一批：`ISO`、`CAB`、`WIM`、`BZ2`、独立 `GZ`、独立 `XZ`、独立 `ZSTD`、`TBZ2`。
- [ ] 第二批：`DEB`、`MSI/Compound`、`NSIS`、`ASAR`、`LZ4`、`LZMA`、`LZ`、`Z`。
- [ ] 第三批：`ALZ`、`EGG`、`ARJ`、`LZH/LHA`、`ACE`、`ZPAQ`、`PEA`。
- [ ] 第四批：`BIN`、`IMG`、`ISZ`、`DAA(1.0)`、`UU/UUE/XXE`、`BH`、`PMA`、`AES`。

优先级不是格式重要性的绝对排序，而是综合了常见程度、现有 Rust/libarchive/helper 生态、测试样本可获得性、跨平台分发风险和对产品定位的影响。
