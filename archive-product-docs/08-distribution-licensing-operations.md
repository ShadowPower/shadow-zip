# 分发、许可证与运维文档

文档日期：2026-05-07

## 1. 分发目标

本产品面向 Windows、macOS 和 Linux 三个平台发布。分发方案必须同时满足用户安装便利性、后端依赖完整性、许可证合规、自动更新可控性和诊断可追踪性。由于项目允许依赖 C/C++ 库和外部 helper，分发设计不能只考虑 Rust 主程序，还必须覆盖动态库、helper 二进制、third-party notices、签名、公证、卸载清理和组件版本检测。

MVP 阶段应优先保证主程序与必要后端可用。shell extension、右键菜单和自动更新可以后置，但安装包从第一天起就应预留文件关联、组件检测和许可证展示能力。

## 2. Windows 分发

Windows 版本需要处理 MSVC 工具链、C/C++ 依赖构建、DLL 搜索路径、安装目录权限、代码签名、文件关联和卸载清理。若后续实现 Explorer 右键菜单，还需要单独设计 shell extension 的注册、升级和卸载流程。

长路径支持是 Windows 版本需要特别关注的事项。归档内路径可能远超传统 MAX_PATH 限制，产品必须在路径处理、解压写入、错误提示和安全检查中统一处理长路径。NTFS 时间戳、权限和备用数据流是否支持，应作为高级兼容项单独评估。

## 3. macOS 分发

macOS 版本应以 app bundle 形式分发，并完成签名和公证。若包含 helper 或动态库，需要确保它们也位于正确的 bundle 位置，并满足签名链要求。对于 Apple Silicon 和 Intel，项目需要选择 universal binary 或分别构建双架构版本。

Finder 集成、文件关联、Dock 最近文件、通知和外部 helper 权限都应纳入 macOS 平台设计。若使用动态库，应确认 rpath 和 bundle 内路径，避免用户移动 app 后后端组件不可用。

## 4. Linux 分发

Linux 版本需要兼顾发行版差异。AppImage 可以降低安装门槛，deb 和 rpm 更适合系统集成，tarball 则适合高级用户。无论采用何种形式，都需要处理 glibc 兼容、动态库依赖、XDG desktop file、MIME association、portal 文件对话框和系统通知。

Linux 文件管理器生态差异较大，Nautilus、Dolphin、Thunar 等扩展机制并不一致。因此，Linux 的右键菜单和深度文件管理器集成不应作为 MVP 阻塞项。更稳妥的策略是先提供文件关联和应用内完整功能，再逐步适配主流文件管理器。

## 5. 第三方依赖与许可证

许可证审计是本项目的发布门禁之一。需要审计的对象包括 Flutter 相关许可证、minizip-ng、sevenz-rust2、libarchive、UnRAR、RAR CLI、7-Zip SDK、7-Zip-zstd、图片解码库、缩放库、EXIF 库以及平台集成库。

RAR 是最敏感的许可证区域。UnRAR 能够支持读取和解压，并不意味着产品可以自由创建 RAR。RAR 创建应依赖官方 rar CLI 或商业授权方案，并在产品功能声明中明确为外部能力。任何将 RAR 创建包装为默认内建能力的方案，都必须先完成授权确认。

静态链接和动态链接也可能带来不同义务。分发包中应包含 third-party notices，并在设置或关于页面提供查看入口。构建系统应能够输出实际打包依赖版本，以便审计和漏洞响应。

## 6. helper 管理

helper 应被视为受控组件，而不是随意调用的外部命令。可能的 helper 包括 7z 兼容 helper、rar CLI、平台 shell helper 和特殊预览 helper。产品启动时可以进行轻量 capability probe，但不能显著拖慢启动。设置页面应显示 helper 路径、版本、能力和可用状态。

调用 helper 时必须使用参数数组，限制工作目录、执行时间、输出大小和环境变量。任务取消时，系统必须终止 helper 并清理临时文件。helper 失败时，错误应被映射为领域错误，而不是直接展示进程退出码。

## 7. 日志与诊断

日志应服务于问题定位，而不是收集用户数据。默认日志可以记录应用版本、平台、后端版本、任务类型、归档格式、codec、filter、错误种类、耗时和资源占用。日志不得记录密码、文件内容或未经处理的敏感路径。

诊断包应由用户主动导出。它可以包含日志、配置摘要、后端 capability、最近错误、性能计数器和缓存统计，但不应包含文件内容、密码或完整目录树。若未来引入自动上传，必须明确征得用户同意，并提供关闭入口。

## 8. 崩溃恢复

崩溃恢复主要服务于长时间压缩、解压和批量任务。任务恢复记录应包含任务 id、任务类型、源归档、目标目录、已完成 entry、失败 entry 和用户策略。恢复时不能盲目继续写入，必须重新检查目标目录、冲突状态和可用空间。

对于解压任务，恢复逻辑应优先保证安全和可理解，而不是追求完全自动。若目标目录已有部分输出，系统应提示用户选择继续、重新开始或打开目标目录检查。

## 9. 自动更新与发布门禁

自动更新应在产品化阶段引入。它必须具备签名校验、通道区分、回滚策略、企业环境禁用和用户可关闭能力。对包含 helper 和动态库的应用，更新过程还必须处理组件版本一致性。

稳定版发布前必须完成三平台构建、样本归档回归、许可证审计、安装与卸载测试、性能基线、安全测试和错误提示检查。正式支持声明必须与测试矩阵一致，不能为了营销扩大承诺范围。
