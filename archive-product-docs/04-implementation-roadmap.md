# 实施路线与验证策略

文档日期：2026-05-07

## 1. 实施原则

本项目应以风险验证为主线安排开发顺序，而不是按界面菜单或格式数量机械推进。真正决定项目成败的风险包括 Flutter 三平台成熟度、C/C++ 后端分发复杂度、7z 扩展 codec 兼容性、tar.xx 单遍流式处理的任务取消能力、图片预览的资源控制，以及大量小文件场景下的 UI 响应能力。只有这些基础能力被原型验证后，产品才适合进入更大范围的功能开发。

实施过程中应坚持两个原则。第一，核心归档能力优先于平台深度集成。右键菜单、文件关联和自动更新固然重要，但它们不应先于 ZIP、7z、tar.zst、任务系统和预览管线。第二，性能基线应与功能同时建立。归档工具最容易出现“功能正确但体验不可接受”的问题，因此每个阶段都需要记录打开时间、解压吞吐、首张预览延迟、峰值内存和临时文件占用。

## 2. 技术验证阶段

技术验证阶段的目标是用最小工程量证明关键路线成立。该阶段不追求完整 UI，也不追求功能覆盖面，而是要确认核心假设是否可靠。

首先，应建立一个 Flutter 三平台窗口原型，并放入虚拟化文件列表。该原型用于验证 Flutter 在 Windows、Linux 和 macOS 上的窗口创建、输入、渲染、大列表滚动和基础布局表现。桌面端原型必须通过 app-core 适配层访问核心能力，尽早验证这一点可以避免后续业务逻辑被 UI 框架变化牵制。

其次，应分别实现三个最小后端原型。ZIP 原型只需要能够读取 central directory 并列出文件；7z 原型必须能够读取 7z + ZSTD 样本，以验证扩展 codec 支持；tar.zst 原型必须以单遍流式方式解压，并确认过程中没有生成中间 tar 文件。与此同时，应实现一个从 ZIP 内 JPEG 读取 metadata 并生成缩略图的预览原型，以验证 entry stream 与图片解码管线的衔接。

该阶段的验收标准很明确：窗口可交互，任务可取消，7z + ZSTD 可读取，tar.zst 无中间文件，图片预览有内存上限，并且所有耗时操作都不阻塞 UI 线程。

## 3. MVP 阶段

MVP 阶段应交付一个可日常使用的主线版本。该版本应支持 ZIP、7z、tar、tar.gz、tar.xz、tar.zst 的打开、浏览、解压和创建，并提供基础图片预览、任务队列、密码输入、冲突处理、缓存管理和结构化错误提示。

MVP 的重点不是覆盖所有格式，而是证明产品工作流完整。用户应能够打开一个 ZIP 或 7z，查看目录，选择文件，预览图片，解压到目标目录；也应能够选择若干文件或目录，创建 ZIP、7z 或 tar.zst。对于 tar.gz、tar.xz、tar.zst，MVP 必须保证单遍流式处理，因为这是产品体验和资源控制的基础要求。

7z + ZSTD、7z + LZ4 和 7z + Brotli 在 MVP 中至少应具备读取能力。创建这些高级 codec 可以先作为实验性或高级选项，但默认创建策略应保持保守，7z 默认使用 LZMA2，ZIP 默认使用 Deflate。这样可以避免用户在默认路径下生成旧工具无法打开的归档。

MVP 不应包含 RAR 创建、shell 深度集成、从归档拖出到系统文件管理器或自动更新。这些功能的工程成本和平台差异较大，不应阻塞核心归档体验。

## 4. 兼容性增强阶段

第二阶段应聚焦复杂归档与真实用户样本。RAR/RAR5 的列表、测试、解压和密码读取应在此阶段完成。tar.xx 索引缓存也应进入正式实现，用于改善顺序格式的二次打开和浏览体验。solid 7z 的单文件预览应通过状态提示、邻近预取、任务取消和进度反馈进行优化。

该阶段还应完善格式 fallback。对于 sevenz-rust2 无法处理的 codec 或 filter，系统应尝试 libarchive 或受控 helper。fallback 不应完全静默，至少应在诊断日志中记录原始后端、失败原因、fallback 后端和最终结果。对于用户可感知的性能变化，界面应显示合理状态，例如正在使用兼容模式打开归档。

兼容性增强阶段的另一个重点是样本归档库。项目应持续收集 ZIP64、AES ZIP、split ZIP、7z + ZSTD、7z + LZ4、7z + Brotli、solid 7z、header encrypted 7z、multipart 7z、tar.zst、RAR5、password RAR、损坏归档、大量小文件、超长路径、符号链接和路径穿越样本。没有样本库，所谓格式支持很容易停留在表面。

## 5. 产品化阶段

第三阶段应处理三平台桌面集成、分发、自动更新和诊断能力。Windows 侧应实现 Explorer 右键菜单、文件关联、长路径处理和签名安装包；macOS 侧应处理 Finder 集成、app bundle、签名、公证和 Dock 行为；Linux 侧应基于 XDG 和 portal 处理文件关联与通知，并根据发行形式选择 AppImage、deb、rpm 或 tarball。

RAR 创建应在此阶段作为外部能力评估。由于 unrar.rs 和 UnRAR 只覆盖读取与解压，不提供 RAR 创建能力，创建 RAR 必须依赖官方 rar CLI 或商业授权方案。产品上应将该能力设计为可检测、可配置、可禁用的外部引擎，而不是默认承诺内建支持。

自动更新和崩溃恢复也适合放在产品化阶段。自动更新必须具备签名校验、通道区分、回滚策略和企业禁用能力。崩溃恢复应记录任务类型、源归档、目标目录、已完成 entry、失败 entry 和用户策略，但不得记录密码或文件内容。

## 6. 测试策略

测试体系应覆盖单元测试、集成测试、端到端测试、安全测试和性能回归。单元测试应验证路径清洗、capability 合并、错误映射、缓存 key、进度聚合和配置迁移。集成测试应覆盖 ZIP、7z、tar、RAR、preview 和 cache 各后端。端到端测试应覆盖打开归档、解压全部、解压选中、创建归档、输入密码、处理冲突、预览图片和取消任务。

安全测试是归档工具不可省略的部分。路径穿越、符号链接逃逸、归档炸弹、巨图解码、外部 helper 参数注入和临时目录逃逸都必须有明确样本和自动化验证。性能测试则应覆盖 SSD、HDD、网络盘、大量小文件、单个大文件、solid 7z 大包、tar.zst 大包和连续图片预览。

每次发布前，应形成一份性能基线报告，至少包括应用启动时间、归档打开时间、listing 构建时间、解压吞吐、压缩吞吐、首次预览延迟、缩略图生成耗时、峰值内存、临时文件占用和取消响应时间。

## 7. 发布节奏

内部 alpha 版本应用于验证架构和三平台可运行性。公共 beta 版本应用于收集真实归档样本、codec 兼容性问题和性能数据。稳定版发布前，必须完成许可证审计、安装卸载测试、主线格式回归、安全测试和错误提示检查。

本项目的版本策略应保持克制。0.x 阶段用于架构验证和 beta；1.0 应代表主线格式稳定、三平台可用、错误模型完整、缓存可控和基础预览可靠；1.x 再逐步扩展 shell 集成、更多格式、高级预览和性能优化。
